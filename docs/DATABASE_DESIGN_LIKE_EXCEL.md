# การทำ Database แบบคิดเป็น Excel (ตาราง/ชีต)



---

## 1. มอง Database เป็น "Excel หลายชีต"

- **1 ตาราง (Table)** = **1 ชีต (Sheet)** ใน Excel  
- **คอลัมน์ (Column)** = **หัวคอลัมน์** ในชีต (เช่น ชื่อ, นามสurname, วันเกิด)  
- **แถว (Row)** = **1 บรรทัดข้อมูล** = 1 เรคอร์ด (record)

ตัวอย่างสั้นๆ:

| วิธีคิดแบบ Excel | ใน Database |
|-------------------|-------------|
| สร้างชีตชื่อ "ลูกค้า" | สร้างตาราง `customers` |
| หัวคอลัมน์: ชื่อ, เบอร์โทร, อีเมล | คอลัมน์: `name`, `phone`, `email` |
| แต่ละแถว = ข้อมูลลูกค้า 1 คน | แต่ละ row = 1 record ในตาราง |

---

## 2. ใช้ Excel/Google Sheets วางแผนก่อนเขียนโค้ด

ก่อนไปเขียน SQL หรือ ORM ในโปรเจกต์ ให้ลอง **ร่างโครงสร้างใน Excel** ก่อน จะได้เห็นภาพชัดและคุยกับเมนเตอร์ง่าย

### ขั้นตอนสั้นๆ

1. **เปิด Excel หรือ Google Sheets**
2. **1 ชีต = 1 ตาราง**  
   - ตั้งชื่อชีตให้ตรงกับชื่อตารางที่จะใช้ใน DB (เช่น `files`, `boards`, `jobs`, `results`)
3. **แถวแรก = ชื่อคอลัมน์ (หัวตาราง)**  
   - พิมพ์ชื่อคอลัมน์ที่ต้องการ เช่น `id`, `name`, `created_at`
4. **แถวถัดไป = ตัวอย่างข้อมูล (ไม่บังคับ)**  
   - ใส่ตัวอย่าง 2–3 แถวให้เห็นภาพว่าข้อมูลจะหน้าตาอย่างไร
5. **ถ้ามีการอ้างอิงข้ามตาราง**  
   - ในชีตนั้นใส่คอลัมน์ที่เป็น "รหัสไปชี้ที่อีกชีต" เช่น ชีต `jobs` มีคอลัมน์ `vcd_file_id` ชี้ไปที่ `id` ในชีต `files`

### ตัวอย่างการร่างใน Excel

**ชีตชื่อ `files`**

| id   | filename   | file_type | storage_path | uploaded_at   |
|------|------------|-----------|--------------|---------------|
| uuid-1 | wave1.vcd | VCD       | /store/xxx   | 2025-02-10 10:00 |
| uuid-2 | fw.erom   | FIRMWARE  | /store/yyy   | 2025-02-10 10:05 |

**ชีตชื่อ `jobs`**

| id   | name   | vcd_file_id | state   | created_at   |
|------|--------|-------------|---------|---------------|
| job-1 | Batch 1 | uuid-1     | pending | 2025-02-10 10:10 |

จากนั้นเวลาไปเขียนโค้ด (SQL หรือ ORM) ก็ใช้โครงนี้เป็น "สเปก" ว่าตารางมีคอลัมน์อะไรบ้าง

## 3. ความต่างสำคัญ: Database vs Excel

| มุม | Excel | Database |
|-----|--------|----------|
| **ประเภทข้อมูล** | แต่ละเซลอาจเป็นตัวเลข/ข้อความปนกัน | แต่ละคอลัมน์กำหนด type ชัดเจน (เช่น integer, varchar, date) |
| **ความซ้ำ** | แถวซ้ำได้ | มักมี **Primary Key** (เช่น `id`) ให้แต่ละแถวไม่ซ้ำ |
| **การอ้างอิง** | ใช้สูตรหรือคัดลอกค่า | ใช้ **Foreign Key** (คอลัมน์ที่ชี้ไป PK ของอีกตาราง) |
| **จำนวนข้อมูล** | มักใช้ไม่เกินหลักหมื่นแถวต่อชีต | DB ออกแบบให้รองรับข้อมูลจำนวนมากและ query ได้เร็ว |

ดังนั้น "ทำ database แบบ Excel" = **ใช้ความคิดแบบตาราง/ชีต** มาออกแบบ DB ไม่ได้หมายความว่าใช้ไฟล์ .xlsx เป็นตัวเก็บข้อมูลจริง

## 4. นำโครงที่ร่างไว้มาใช้ในโปรเจกต์ Eval System

ในโปรเจกต์นี้ โครงสร้าง DB อยู่ที่ `backend/db/orm_models.py` และมี 4 ตารางหลัก:

- **files** — ลงทะเบียนไฟล์ที่อัปโหลด  
- **boards** — รายการบอร์ดและสถานะ  
- **jobs** — คิวงานทดสอบ (มี FK ไป `files`)  
- **results** — ผลรันทดสอบ (มี FK ไป `jobs`, `boards`)

ถ้าเมนเตอร์ให้ "ทำ database" เพิ่ม (เช่น เพิ่มตารางหรือคอลัมน์) แนะนำให้:

1. **ร่างใน Excel ก่อน**  
   - สร้างชีตใหม่ (หรือเพิ่มคอลัมน์ในชีตเดิม)  
   - ตั้งชื่อคอลัมน์และใส่ตัวอย่าง 2–3 แถว  
2. **เช็กความสัมพันธ์**  
   - ถ้าตารางใหม่ต้อง "ชี้ไปที่" ตารางเดิม ให้มีคอลัมน์แบบ `xxx_id` (Foreign Key)  
3. **นำไปเขียนในโค้ด**  
   - เพิ่มคลาสใน `orm_models.py` (หรือสร้างไฟล์ model ใหม่แล้ว import)  
   - สร้าง migration ถ้าโปรเจกต์ใช้ Alembic  

ถ้าต้องการรายละเอียดโครงสร้างแต่ละตารางแบบเต็ม อ่านได้ที่ `docs/DATABASE_STRUCTURE.md`

## 5. สรุปสั้นๆ

- **Database แบบ Excel** = คิดโครงสร้าง DB เป็น **ตาราง (ชีต)** มี **คอลัมน์** และ **แถว**  
- **ใช้ Excel/Sheets วางแผน** = 1 ชีตต่อ 1 ตาราง, แถวแรกเป็นชื่อคอลัมน์, แถวถัดไปเป็นตัวอย่างข้อมูล  
- **เมื่อทำ database จริง** = นำโครงที่ร่างไว้ไปเขียนเป็นตารางใน DB (SQL/ORM) และกำหนด type, Primary Key, Foreign Key ให้ชัดเจน  

ถ้าเมนเตอร์ให้โจทย์เฉพาะ (เช่น "เพิ่มตารางเก็บ test case" หรือ "เพิ่มคอลัมน์ใน jobs") สามารถร่างเป็นชีต Excel ตามด้านบนแล้วค่อยแปลงเป็นโค้ดในโปรเจกต์ได้เลย
